# Browxai Report - 2026-05-20 - Codex

## Context

Used Browxai from Codex against Clipro's authenticated local SPA to verify a media-editor timeline workflow:

- Opened a persistent profile session.
- Restored the Clipro timeline sidebar and AI Voiceover tab.
- Used `drag` to reorder timeline clips for replay edge-case testing.
- Used `eval_js`/`poll_eval` to inspect Redux state and upload CSV files through the existing hidden file input.

## Items to Flag

### `drag` Capability Gate Was Confusing Before Restart

Before the MCP restart, `get_config({ scope: "resolved" })` reported `unstable`, but `drag` still returned:

```text
tool "drag" is disabled (capability not in BROWX_CAPABILITIES)
```

The root cause was understandable after reading the runbook: the server-level capability gate is resolved once at process startup, while config inspection can show newly persisted config that has not affected the live gate. This is technically correct, but the operator experience is confusing.

Wishlist:

- Include an `effectiveStartupCapabilities` field in `get_config`, or
- Have disabled-tool errors mention that `set_config` changes require MCP server restart when the resolved persisted config already includes the capability.

### `point_probe` Failed on Timeline Coordinates

Calling `point_probe` against timeline coordinates repeatedly failed with:

```text
Cannot read properties of undefined (reading 'stack')
```

I worked around it with `eval_js(() => document.elementsFromPoint(...))`, which produced the information I needed. `point_probe` would have been the right primitive for this media-editor timeline.

Wishlist:

- Make `point_probe` return a structured failure with the coordinate and page URL.
- Preserve partial `elementsFromPoint` data if the crop/stack formatting path fails.

### Drag Targeting Can Accidentally Resize Tiny Timeline Items

Dragging a very narrow replay clip near its edge resized it instead of reordering it. This is application behavior, not necessarily a Browxai bug, but timeline/media-editor testing needs sharper diagnostics around what the press point hit.

Wishlist:

- A drag preflight that reports the top hit element and nearest draggable/resize handles before moving.
- An optional `startOffset` or `avoidEdges` helper for element-target drags so agents can reliably press the center of draggable content, not a resize handle.

### File Upload Still Requires `eval_js`

CSV upload verification required injecting a `File` and `DataTransfer` into a hidden file input through `eval_js`. That worked, but it is a common browser-testing primitive and would be safer as a first-class tool.

Wishlist:

- Add a `set_input_files`/`upload_file` action tool that accepts a selector/ref plus file name, MIME type, and content/path.

### Tool Schema Oddity

The exposed `drag` schema displayed `to.coords` as `string`, while the tool accepted a normal `{ x, y }` coordinate object. This did not block the run, but it is easy for an agent to misread.

Wishlist:

- Align the MCP schema for `drag.to.coords` with `drag.from.coords`.

## What Worked Well

- Persistent profile restored authenticated Clipro state cleanly after restart.
- `drag` worked reliably once the startup capability gate included `unstable`.
- `poll_eval` was very effective for waiting on Redux state after CSV uploads.
- `screenshot` with `scale: "css"` and JPEG quality was useful for quick UI confirmation without excessive payload.
