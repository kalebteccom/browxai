# RFC 0004 — Architecture hardening: a safety-critical maintainability standard for browxai

**Date:** 2026-06-14
**Status:** Draft — proposal. No code changed by this RFC; it is the design record + the standard + the phased plan. The companion references under [`references/`](references/) carry the evidence, the target patterns, the fitness-function specs, the AI-doc/harness changes, the prior art, and the future-proofing roadmap.
**Author:** Claude (adversarial audit across 8 subsystems + synthesis, June 2026)
**Trigger:** Owner directive — _"deep adversarial analysis of what is currently spaghetti, what can be improved for scalability and maintainability, especially SOLID / open-closed across all files; a full refactor into good standards; the AI documentation and harness hardenings to add the guardrails that prevent going back to a poor codebase; go far beyond the existing principles — NASA-level maintainability."_ Extends and **operationalizes** [`architecture-principles.md`](../ai-context/architecture/architecture-principles.md) and [`code-quality.md`](../ai-context/agent-process/code-quality.md); continues the seam work of [RFC 0002](0002-multi-engine-bidi.md) (the `BrowserEngine` port) and [RFC 0003](0003-capability-ports-decoupling.md) (the capability substrates).

---

## 0. The one-paragraph thesis

browxai's architectural **doctrine is excellent and the seams are mostly real** — ports-and-adapters at the engine boundary, capability substrates, a `Transport` abstraction, a dependency-inverted `PluginApi`, a 382-line composition-root `server.ts`. The doctrine is **not the problem.** The problem is that **the doctrine is unenforced**: nothing in CI, lint, or the test suite _fails_ when a developer (human or agent) violates it. An adversarial audit of all 8 subsystems ([`references/0004-01-current-state-audit.md`](references/0004-01-current-state-audit.md)) found **80 concrete SOLID/spaghetti defects**, and they cluster around a single failure mode — **the architecture drifted exactly where no machine was watching.** The headline: the project's own flagship claim — _"a new engine is a new adapter behind the existing port"_ (architecture-principles §4) — **is false today.** A sixth engine requires editing **5–8 existing files** because the adapter _wiring_ is hardcoded `if (engine === "literal")` across three session factories, the session registry's 18 scattered Safari guards, five substrate selectors, and the deep-tool gate. The fix is two-pronged and inseparable: **(1)** a focused, behavior-preserving refactor to **registry- and metadata-driven extension points** (close OCP) plus **module/object segregation** (SRP/ISP) plus **shared-pattern extraction** (DRY); and **(2)** a **safety-critical maintainability standard** — adapted from the practices that keep flight software alive (NASA/JPL _Power of Ten_, the JPL Institutional Coding Standard, DO-178C traceability, defense-in-depth, bounded-resource discipline) — in which **every architectural invariant is backed by an executable fitness function, custom lint rule, or CI gate**, so the codebase _physically cannot_ decay back to its current state.

## 1. Why "NASA-level," and what it means here

The owner asked to go _far beyond_ the existing principles toward NASA-level maintainability. That is a precise, citable target, not a slogan. Safety-critical software (avionics, spacecraft, medical) does not stay maintainable because its engineers are more disciplined; it stays maintainable because **discipline is mechanized**. The relevant bodies of practice — surveyed in [`references/0004-07-prior-art-and-references.md`](references/0004-07-prior-art-and-references.md) — are:

- **The Power of Ten** (G. Holzmann, JPL/NASA LaRC, 2006): ten rules for safety-critical code, every one mechanically checkable — bounded loops, no unbounded recursion, no dynamic allocation on the hot path, small functions, **assertion density ≥ 2 per function**, smallest-scope data, **check every return value**, compile clean with all checkers at maximum.
- **The JPL Institutional Coding Standard** (rules layered by severity, each tied to a static analyzer).
- **DO-178C / NASA NPR 7150.2** traceability: every requirement traces to code traces to a test; **coverage is a structural property, not a hope.**
- **Defense in depth / fault containment**: every boundary validates; a fault is _contained_ and surfaced as a structured refusal, never a crash — browxai's anti-wedge deadline + structured-refusal pattern is already this; we generalize it.
- **Evolutionary architecture / fitness functions** (Ford, Parsons, Kua, 2017): an architectural characteristic you care about gets an **automated test that fails when the characteristic regresses.** This is the load-bearing import for browxai.

The adaptation to a TypeScript MCP server is laid out in full in [`references/0004-02-maintainability-standard.md`](references/0004-02-maintainability-standard.md). The standard is summarized as **ten browxai laws** in §4. The crucial reframing: in C, the Power-of-Ten rules guard against _memory corruption and undefined behavior_; in browxai, the analogous failure mode is **architectural corruption** — a god-module, an `if (engine === …)` leak, a hand-edited central list — and the analogous mechanization is the **fitness function**.

## 3. Scope and non-goals

**In scope:** a behavior-preserving refactor of the extension seams (engine, capability, tool, substrate, transport, CLI) to registry/metadata-driven OCP; segregation of the god-modules and god-objects; extraction of the copy-paste families; the maintainability standard; the full guardrail/fitness-function layer; the AI-documentation + harness changes that encode and enforce it; the prior-art record; and the future-proofing roadmap.

**Non-goals (explicit):** No new user-facing features. No behavior change — every refactor step is byte-identical per engine, verified by the existing 1834 unit + five-engine keystone suites (the same strangler-fig discipline RFC 0002/0003 used). No speculative ports: per architecture-principles §1, an abstraction lands only where there is a **second real implementation or a committed near-term need** — and the audit confirms every proposed registry has _multiple existing implementations today_ (5 engines, 70+ batchable tools, 9 perf analysers, 3 transports, 5 CLI commands), so none are speculative. We do **not** rewrite the engine adapters, the substrates, or the plugin runtime — those are the parts that are _right_; we fix their _wiring_.

## 4. The standard, in ten laws

The full standard with rationale, the safety-critical lineage of each law, and the exact enforcement mechanism is [`references/0004-02-maintainability-standard.md`](references/0004-02-maintainability-standard.md). The ten browxai laws, each with its **mechanized enforcer** (the column that does not exist today — that is the work):

| Law                                         | Statement                                                                                                                                                                                     | Lineage                                                                 | Enforcer (the fitness function / gate)                                                                                                    |
| ------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------- |
| **L1 — Closed core**                        | No module above the engine seam may name an engine, a transport, or a concrete adapter. Extension is add-only.                                                                                | Power-of-Ten "simple control flow"; OCP                                 | `no-engine-literal-branches` lint rule + the **engine-adapter-contract** keystone (a mock 6th engine that must work with zero core edits) |
| **L2 — Single source of truth**             | No fact is written twice. Capabilities, batchability, deep-ness, tool-types, layer precedence are **declared once** at the unit and **derived**, never hand-listed.                           | DRY; DO-178C config-data discipline                                     | **completeness fitness tests** (every registered tool ∈ derived capability map; tool-types ≡ schemas) + codegen                           |
| **L3 — One reason to change**               | One module, one responsibility; one object, one role. Hard budgets: a tool module ≤ ~400 LOC, `server.ts` ≤ 400, a function ≤ ~70 LOC / complexity ≤ ~15.                                     | Power-of-Ten rule 4; SRP; Parnas                                        | `eslint` `max-lines` / `complexity` / `max-lines-per-function` budgets + the **composition-root guard**                                   |
| **L4 — Segregated contracts**               | No god-object. Consumers depend on the narrow port they use, not a 75-member bag.                                                                                                             | ISP; Lakos levelization                                                 | interface-member budget + dependency-cruiser "ToolHost split" rule                                                                        |
| **L5 — Substitutable adapters**             | Every adapter honors its port's full contract or **declares the gap as a capability**; no adapter throws where the port promises a value.                                                     | LSP; defense-in-depth                                                   | the **port-conformance** contract test run against _every_ adapter incl. a synthetic one                                                  |
| **L6 — Validate at the edge, trust within** | Untyped data is narrowed at the system boundary (MCP wire, config, CDP/Playwright edge) and is **fully typed thereafter**.                                                                    | Power-of-Ten "check returns"; the just-landed `no-unsafe-*` enforcement | the 5 `no-unsafe-*` rules (now `error`) + `no-explicit-any` + the boundary-narrowing test                                                 |
| **L7 — Bounded everything**                 | Every loop, buffer, ring, recursion, and wait has an explicit, tested bound. No unbounded fan-out, slurp, or recursion.                                                                       | Power-of-Ten rules 2–3; architecture-principles §3                      | the **bounded-resource** lint rule + budget tests on rings/deadlines/depth caps                                                           |
| **L8 — Assert the invariants**              | Internal invariants are asserted, not assumed; a violated invariant surfaces as a structured refusal, never a crash or silent wrong answer.                                                   | Power-of-Ten rule 5 (assertion density); fault containment              | an `invariant()` helper + a density check on the load-bearing modules                                                                     |
| **L9 — Traceable**                          | Every world-touching tool ⇒ a capability declaration ⇒ a keystone denial test, **in the same change**. Every engine ⇒ a capability row ⇒ a keystone lane. Coverage of the seam is structural. | DO-178C/NPR-7150.2 traceability                                         | **traceability fitness tests** (tool↔capability↔keystone; engine↔caps↔lane)                                                               |
| **L10 — Deterministic & observable**        | The surface is deterministic where it pays (replay, diffing) and self-diagnosing (the diagnostics recorder); determinism is keystone-verified, not asserted.                                  | DO-178C reproducibility; architecture-principles §3                     | the existing keystone determinism gates, extended to the new seams                                                                        |

Each law's enforcer is specified, with concrete config and code, in [`references/0004-05-fitness-functions-and-guardrails.md`](references/0004-05-fitness-functions-and-guardrails.md). **The standard is not aspirational prose — every law above is a green check or it is not in the standard.**

## 5. Decisions

Each decision is one architectural ruling, tied to the audit evidence and a target pattern. The full pattern catalogue with before/after code is [`references/0004-03-ocp-registry-patterns.md`](references/0004-03-ocp-registry-patterns.md); the sequenced plan is [`references/0004-04-refactor-plan.md`](references/0004-04-refactor-plan.md).

<a id="d1"></a>**D1 — Engine extension closes behind an `EngineRegistry`.** The five adapters are already well-isolated classes; their _instantiation and post-creation wiring_ are not. Replace the triplicated `if (engine === …)` chains in `managed.ts`/`incognito.ts`/`byob.ts`, the 18 scattered Safari guards in `session-registry`, and the five `*For(e)` substrate selectors in `host-build.ts` with **one `EngineRegistry`**: each engine registers, in _one_ file, a record `{ kind, capabilities, makeAdapter, makeSubstrates, postWire }`. The factories, selectors, and gate become **data-driven lookups** keyed by `session.engine`. A sixth engine becomes a **single new file** + one registration line. This is the literal realization of architecture-principles §4 ("new engine = new adapter"), which is false today. Enforced by L1 (`no-engine-literal-branches` + the engine-adapter-contract keystone).

<a id="d2"></a>**D2 — Tools self-declare metadata at registration; central lists are derived.** A tool today must be hand-added to up to five disjoint lists (`TOOL_CAPABILITY`, `BATCH_ALLOWED_TOOLS`, `DEEP_TOOLS`, the SDK types, the docs), and every miss is silent (capability silently defaults to `human`; a batchable tool silently is not). Replace this with **colocated declaration**: `host.register(name, { capability, batchable, deep, … }, schema, handler)`. The central maps (`TOOL_CAPABILITY`, the batch allow-set, the deep gate, the SDK tool-types) are **derived from the registrations**, not hand-maintained. Enforced by L2 (completeness fitness tests + tool-types codegen). This single change closes four of the audit's OCP findings at once.

<a id="d3"></a>**D3 — Split the god-modules; segregate the god-objects.** The coarse `src/tools/*` modules (an artifact of [RFC 0003](0003-capability-ports-decoupling.md)'s line-range decomposition, explicitly flagged as "thematically loose" at the time) violate the doctrine's own _one-tool-one-file_ rule. Split by cohesive concern toward one-family-per-module (and, where the family is itself heterogeneous like `emulation-config`, by domain: emulation / config / approvals / secrets / captcha). Segregate `ToolHost` (75 members) into composable sub-ports a handler depends on à la carte (`GateHost`, `SessionHost`, `ActionHost`, …) and `SessionEntry` (50+ fields) into the role-bundles its consumers actually use. Enforced by L3 (size/complexity budgets) and L4 (interface-member budget).

<a id="d4"></a>**D4 — Collapse the copy-paste families to shared abstractions.** Five concrete patterns repeat: (a) the five policy classes (dialog/permission/notification/fs-picker/device-emu) share an identical buffer+record capture — extract a `PolicyBuffer<T>` base; (b) the five substrate selectors are identical boilerplate — fold into the D1 `EngineRegistry`; (c) the 7-step action handler pattern repeats ~50× — extract an `actionTool()` wrapper; (d) egress masking (URL-sanitiser + secrets) is hand-called at each sink — introduce one `EgressSanitiser` chokepoint injected into every output path; (e) `sample.ts`/`network.ts`/`extract.ts` internal duplication — extract the shared scanner/folder. Enforced by L2 (the egress chokepoint becomes a _compile-time_ guarantee, not a discipline) and a `jscpd` duplication budget in CI.

<a id="d5"></a>**D5 — Close the Safari LSP leak: the substrate ports are the only contract.** `BrowserSession.page()` throwing on Safari (forcing 18 scattered defensive guards) is a Liskov violation that leaks the no-Page seam into every caller. Resolution: callers **never** reach for `page()` — they go through the capability substrates (the RFC 0003 ports), which is _already_ the design intent; the residual direct `page()` calls are the bug. Make `page()` a capability the engine _declares_ (`session.page` present only when the engine has one), typed so a caller that needs it must narrow, and move the 18 guards into the single `EngineRegistry.postWire`. Enforced by L5 (the port-conformance test forbids a port method that throws unconditionally) and L1 (no caller may branch on the absence).

<a id="d6"></a>**D6 — Replace every extensibility `switch` with a registry.** The CLI subcommand switch, the SDK transport switch, the perf `ANALYSERS` stringly-map, the session-mode literals, and `PM_VERBS` are all OCP violations with multiple real cases today (proven seams). Each becomes a `Map<key, factory>` registered add-only. The perf-audit analyser registry is _already_ the doctrine's cited OCP exemplar (architecture-principles §2) — the audit found its **implementation** is stringly-typed and edited per category; D6 makes the exemplar actually exemplary.

<a id="d7"></a>**D7 — Generate the SDK tool-types; never hand-mirror.** `sdk/tool-types.ts` (673 LOC) hand-mirrors the zod schemas it admits are the source of truth — guaranteed drift. A build-time codegen reads the registrations (post-D2 they carry their schemas) and emits the types; a fitness test fails if the committed file diverges from the regenerated one. (This composes with the type-safety work already landed: handler args are now `z.infer` of the inputSchema — the SDK types should be the same inference, generated.)

<a id="d8"></a>**D8 — Adopt the maintainability standard (§4) as binding.** The ten laws become part of the gate. This is the doctrine upgrade the owner asked for: the existing `architecture-principles.md` is _retained and extended_ (not replaced) — its §7 review checklist gains the ten laws and their enforcers; full text in [`references/0004-06-ai-documentation-and-harness.md`](references/0004-06-ai-documentation-and-harness.md).

<a id="d9"></a>**D9 — Every architectural invariant gets an executable fitness function.** This is the heart of the hardening and the answer to T7. The fitness-function suite (`test/architecture/**`) is a first-class part of `pnpm test`: OCP regression tests (mock-engine contract, capability extensibility), completeness tests (tool↔capability↔batch↔deep↔types↔keystone), the port-conformance contract test, the bounded-resource and assertion-density checks, the determinism gates. Specs + code in [`references/0004-05-fitness-functions-and-guardrails.md`](references/0004-05-fitness-functions-and-guardrails.md).

<a id="d10"></a>**D10 — Enforce the dependency graph (levelization).** Introduce `dependency-cruiser` (or an equivalent custom check) encoding the layering the doctrine _asserts_ but no machine _checks_: `server.ts`/`tools/*` may not import `sdk/*` or `cli/*`; `page/*` handlers may not import a concrete engine adapter or a transport; `sdk/*` may not import handler internals; nothing imports `cli/*` except the bin; the core never imports outward. This is Lakos-style levelization made executable — the single highest-leverage guardrail against DIP rot.

<a id="d11"></a>**D11 — Budgets, not vibes.** File-size, function-length, cyclomatic-complexity, interface-member-count, and duplication budgets are `error`-level lint, sized from the _current healthy_ modules (not the god-modules). The composition root has a hard 400-line ceiling. Budgets are how L3/L4/L7 stay true after the refactor lands — they are the ratchet.

<a id="d12"></a>**D12 — The AI documentation + harness encode and enforce the standard.** The guardrails must be _discoverable by the next agent_ and _exercised by the harness_. `code-quality.md` gains an "Architecture enforcement" section listing every guardrail and its check; `architecture-principles.md` gains the ten laws; a new `docs/ai-context/architecture/fitness-functions.md` is the index of executable invariants; the agent skills gain an `architecture-fitness-auditor`; the RFC index and `repo-map.md` are corrected (the audit found the coupling-audit reference still cites `server.ts` at 12,889 lines). Full diff in [`references/0004-06-ai-documentation-and-harness.md`](references/0004-06-ai-documentation-and-harness.md).

## 6. Phasing (strangler-fig; gate green every step; guardrail-first)

The sequencing is deliberate and inverts the usual instinct: **the guardrails land first**, so the refactor is performed _against_ the fitness functions that will keep it true — and so the refactor's own behavior-preservation is mechanically verified. Full per-phase file lists, risk, and rollback in [`references/0004-04-refactor-plan.md`](references/0004-04-refactor-plan.md).

- **P0 — Instrument before operating.** Land the fitness-function harness (`test/architecture/**`), `dependency-cruiser`, and the size/complexity/duplication budgets **as reporting-only** (warn), plus the completeness/traceability tests **as failing** (they pass today for the _existing_ maps — they just freeze them). Add the `no-engine-literal-branches` and `no-inlined-capability-checks` lint rules scoped to _new_ violations. This phase changes no `src/` behavior; it makes the invariants visible and stops the bleeding.
- **P1 — The `EngineRegistry` (D1, D5).** The single highest-leverage refactor. Collapse the three factory chains + 18 registry guards + 5 selectors into the registry; close the Safari LSP leak. Promote `no-engine-literal-branches` to `error`. The engine-adapter-contract keystone goes green. **A sixth engine is now one file.**
- **P2 — Metadata-at-registration (D2, D7).** Colocate `{capability, batchable, deep}` at `host.register`; derive the central maps; generate the SDK types. Promote the completeness tests from "frozen" to "derived." Four OCP findings close.
- **P3 — Module/object segregation (D3, D4).** Split the god-modules to the size budget; segregate `ToolHost`/`SessionEntry`; extract `PolicyBuffer`, `actionTool()`, `EgressSanitiser`. Promote the size/complexity/duplication budgets to `error`.
- **P4 — The remaining switches (D6) + dependency layering (D10) to `error`.** CLI/transport/analyser/mode registries; turn the dependency-cruiser rules from warn to error.
- **P5 — Assertion density + bounded-resource pass (L7, L8) + the doc/harness landing (D12).** Add `invariant()` and the density check on the load-bearing modules; the bounded-resource audit; ship the `architecture-principles.md` / `code-quality.md` / `fitness-functions.md` / skill updates.

Each phase is independently shippable and gate-green; no phase changes external behavior; the five-engine keystones are the regression gate throughout.

## 7. Alternatives considered

- **Do nothing / code-review only.** Rejected: T7 is the proof that review does not hold the line — every one of the 80 defects passed review. Human review is necessary but demonstrably insufficient against architectural drift over an agent-driven, high-velocity codebase.
- **A big-bang rewrite of `src/tools` / `src/session`.** Rejected: violates the strangler-fig discipline that made RFC 0002/0003 safe; the adapters and substrates are _correct_ — the defect is their wiring. A rewrite risks behavior change for no architectural gain the registries don't already deliver.
- **Speculative maximal abstraction (a port for everything).** Rejected by architecture-principles §1 (the proven-seam test). The audit was used precisely to confirm every proposed registry has _multiple real implementations today_; none are speculative. Where there is a single implementation (e.g. one workspace today), we keep the concrete code and add the seam later.
- **Adopt an off-the-shelf "clean architecture" framework.** Rejected: browxai's seams are domain-specific (engine capability, MCP wire, page-side functions); a generic framework adds indirection without fitting the domain. We import the _practices_ (Power-of-Ten, fitness functions, levelization), not a framework.

## 8. Risks and mitigations

- **Refactor introduces behavior change.** Mitigated by strangler-fig + the five-engine keystones + the 1834 unit suite as the regression gate on every phase, and by P0 landing the fitness functions _first_ so each subsequent phase is verified against frozen invariants.
- **Guardrails become noise / are disabled.** Mitigated by sizing budgets from the _healthy_ current modules (not aspirational), landing as warn before error, and the meta-rule (L-meta) that a guardrail may only be relaxed via an RFC amendment with rationale — never an inline disable (the `no-unsafe-*` work already established this norm).
- **The fitness suite is slow.** Mitigated: the architecture tests are static (AST/graph/string analysis, no browser) and run in the fast `pnpm test` lane, not the keystone lane; the mock-engine contract test is the only one that touches the runtime and it uses an in-memory fake.
- **Agent velocity drops.** The opposite is the intent and the likely outcome: a hardened seam makes the common extension (new tool, new engine, new transport) _add-only_, which is faster and lower-risk for an agent, and the fitness functions give an agent immediate, local feedback instead of a deferred review.
