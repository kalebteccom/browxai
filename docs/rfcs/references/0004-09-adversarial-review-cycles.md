# RFC 0004 / Reference 09 — Adversarial review cycles

**Parent:** [`0004-architecture-hardening.md`](../0004-architecture-hardening.md) — this document is the review record for the RFC 0004 suite (the spine plus references `0004-01`…`0004-08`).
**Status:** Reference — **converged**. The iterate-until-clean adversarial review terminated at **Cycle 10** with a dual-clean return (both reviewers, zero material defects; trajectory 72 → 7 → 8 → 5 → 9 → 7 → 6 → 1 → 1 → 0). No code changed by this document; it records how the design docs were stress-tested against the real source and what each of the ten cycles corrected.

A design document that asserts file:line evidence is only as trustworthy as its weakest citation — and an eight-subsystem audit synthesised by an automated fan-out is exactly the kind of artifact that accrues confident-but-wrong numbers. So the RFC 0004 suite is not merely written and reviewed once; it is held to an **iterate-until-clean adversarial loop**: independent reviewers re-derive every load-bearing count and citation from the live tree, the fix list is reconciled against verified ground truth, and the loop repeats until a cycle returns zero material defects. This reference is the audit trail of that loop — the method, the per-cycle verdicts, and the disposition of every fix.

---

## The method

Each cycle runs **two independent adversarial reviewers in parallel**, deliberately chosen to be heterogeneous so their blind spots do not coincide:

- **Reviewer A — Opus at maximum rigor** (general-purpose, model `opus`): a full read of the spine and all eight references against the source tree, instructed to assume every number is wrong until it is re-derived from the file it cites.
- **Reviewer B — Codex at its strongest model and maximum reasoning effort** (`codex-rescue`): a second, independently-prompted pass that re-opens the same citations and additionally type-checks every code sketch against the real exported symbols.

Both reviewers verify against the **real source**, not against each other and not against the prose's own claims: a count like "`DEEP_TOOLS` has _N_ names" is re-derived by reading `src/engine/tool-gate.ts`, not by trusting the audit. The orchestrator then **reconciles** the two defect lists against a single verified ground-truth set (the orchestrator re-checks any disagreement against the source, because the reviewers themselves sometimes disagree), produces one triaged fix list, and the fixer applies it. The loop **repeats** — a fresh cycle re-reviews the corrected suite — and terminates only when a cycle returns **zero material defects**. Strong convergence between two heterogeneous reviewers on the same defect is treated as high-confidence signal; a defect only one reviewer raises is still verified against source before it enters the fix list.

The bar for "material defect" is a claim a skeptical staff engineer could falsify by opening the cited file: a wrong count, a fabricated line range, a code sketch that references a symbol that does not exist, an external citation that misattributes a source, or an internal contradiction between two references. Stylistic notes are out of scope; this loop exists to make the evidence floor unfalsifiable.

## Cycle 1

**Inputs.** The spine and references `0004-01`…`0004-08` as first drafted. **Outcome:** both reviewers returned material defects; the suite was corrected and a Cycle 2 re-review scheduled.

### Verdicts

- **Reviewer A (Opus, max rigor) — 28 defects found.** Dominant themes: the `ToolHost` "75 members" drift propagated across all nine docs (the real interface declares **33**); the session-registry Safari guards quoted as "18"/"18+" where the source has **17**; `DEEP_TOOLS` cited *three different ways* across the suite (26, ~37, and the true **31**); the perf `ANALYSERS` count quoted as 9 where the source has **8**; and the logical overclaim that the codebase _"physically cannot decay back to its current state"_ — which 0004-02's own §3 L1(f) concedes the lint rule can be satisfied by an incomplete registry.
- **Reviewer B (Codex, strongest model + max reasoning effort) — 44 defects found.** A superset of A's number drifts plus ~10 **code-sketch symbol errors** the type-check pass surfaced: the `SessionMode` and `EngineNotYetSupportedError` imports, the British `sanitiseUrl`/`./url-sanitiser.js` spelling (the real export is `sanitizeUrl` from `./url-sanitizer.js`), the missing `NetworkEntry` import, an undefined `TransportMode`, a non-existent `api.z` on `PluginApi`, the `isToolEnabled` arity (2-arg today, not 3), a `server.registeredSchemas()` method that `createServer` does not expose, and an `EngineRegistry` import used in a P0 test that does not exist until D1. Reviewer B also caught the **engine-registry contract drift** (three different record shapes across docs — `makeSession` vs `makeAdapter`, `EngineRecord` vs `EngineDefinition` vs `EngineEntry`), the **EgressSanitiser / port-conformance no-op-guarantee** logic gaps, the `depcruise`/`depgraph` script-name drift, the **P0-failing-test vs gate-green** contradiction, and the **bounded-resource warn/error** mismatch between 0004-04 and 0004-05.

**Convergence.** The two reviewers — heterogeneous in model and prompt — converged strongly on the systemic number drifts (75→33, 18→17, DEEP_TOOLS→31, 9→8), which the orchestrator therefore treated as high-confidence and verified once against source as the canonical values. Codex's larger count came almost entirely from the code-sketch symbol layer Opus did not type-check, not from disagreement on the numbers. Combined and reconciled against verified ground truth, the cycle produced the triaged fix list dispositioned below.

### Headline systemic defects

1. **`ToolHost` "75 members" everywhere → 33.** The single most-propagated drift; appeared in the spine, 0004-02, -03, -05, -06, -07, -08. The real interface (`src/tools/host.ts:54-189`) declares 33 members. In 0004-05 §4 this also forced a **re-baselined budget**: a "freeze at 75, target ≤40" ceiling is vacuous because the real value (33) already passes ≤40, so the budget is frozen at the real 33 and the meaningful constraint moved to the post-split sub-ports (≤~12 each).
2. **Safari guards "18"/"18+" → 17.** The enumerated guard line list (`session-registry.ts` `:266,280,292,301,332,338,349,383,408,441,451,457,479,536,550,584,589`) is 17 lines; 0004-02 §3 additionally carried *fabricated* sub-ranges (`301-314`, `349-370`, `408-432`, …) replaced with the 17 real single-line numbers.
3. **`DEEP_TOOLS` cited three ways → 31.** The audit said 26; 0004-01's own ledger then "corrected" it to ~37; the real set (`src/engine/tool-gate.ts:38-88`) is 31 entries. Every occurrence — including the ledger's own correction — was set to 31.
4. **perf `ANALYSERS` "9" → 8.** Eight categories (render-blocking, unused-code, oversize-images, layout-thrashing, long-tasks, leak-suspects, cache-opportunities, font-loading) at `src/page/perf-audit.ts:88-97`.
5. **`BATCH_ALLOWED_TOOLS` "72" → 71.** The Set (`src/tools/host-build.ts:640-712`) holds 71 entries; the spine's looser "70+" is fine and was kept.
6. **The "physically cannot decay" overclaim → softened.** Recast to "cannot decay through a green gate without a visible, reviewed amendment," which is what the guardrails actually guarantee.
7. **Engine-registry contract drift → one shape.** Standardised to `EngineEntry` `{ kind, capabilities, makeAdapter, makeSubstrates, postWire }` across every doc; `byob.ts` described as one of "three engine-literal dispatch sites" (it uses attach/refusal paths) rather than one of "three identical chains."
8. **External-citation accuracy.** Holzmann's _Power of Ten_ re-attributed to **NASA/JPL (Laboratory for Reliable Software)** (not "JPL/NASA LaRC"); rule 3 ("no dynamic memory allocation after initialization") marked as browxai's **adaptation** ("no unbounded allocation on the hot path"), not a clean import; rule 2 (bounded loops) noted as the direct import; and the L8 phrasing that "an assertion that fails takes a defined recovery action" softened — assertions are side-effect-free and failure handling is defined at the system level, not per-assertion.
