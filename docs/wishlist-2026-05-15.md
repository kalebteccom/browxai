# Wishlist — beyond the round-8 adoption report

**Date:** 2026-05-15
**Companion to:** [`adoption-report-2026-05-15.md`](adoption-report-2026-05-15.md) (the canonical Phase-1 exit report) and the consumer-workspace `RUNBOOK-TEST-DRIVE-REPORT.md` (rounds 1–7 of site-docs use against the target app — that report lives outside this repo, in the consumer's workspace).

The adoption report names what *blocked* (or almost blocked) the re-adoption run — five items, three 🟡, two 🟢. This document is the **forward-looking wishlist**: things that didn't block any single run but which, accumulated across all 8 rounds, made the loop slower or chunkier than it needs to be. Organised by what they buy.

---

## A. Context-usage reductions (the biggest category)

The largest token sinks across this session, in rough order: (1) snapshots dumping the whole interactive tree when I wanted one subtree, (2) full Playwright actionability stack-traces in halt-error logs, (3) console-error blocks containing 50-line React component stacks, (4) repeated `Bash` outputs from throwaway Playwright-over-CDP scripts that existed only because there was no eval primitive.

### A1. Scoped snapshots — `snapshot({ scope: <ref>, maxNodes: N, omit: [<patterns>] })`

The target-app snapshot routinely returned 200–500 nodes; on most calls I needed 5. Three knobs would change that:

- **`scope: <ref>`** — dump only the subtree rooted at a previously-observed ref. "Just the the AI feature side-panel under e205" instead of 500 entries.
- **`maxNodes: N`** + a `truncated: true` flag — bounded dump, like `head -N` for a tree.
- **`omit: ["timeline-segment-*", "clip-thumbnail"]`** — pattern-omit known-noisy regions. target-app's timeline area was ~80% of my snapshot tokens and I never queried into it; a single omit pattern would have halved every snapshot.

Closes a real measurable cost: each full target-app snapshot was ~7–10k tokens; a scoped one would be ~500–1500.

### A2. `snapshotDelta` modes that actually scope

The runbook documents `mode: "scoped_snapshot" | "tree_diff"` on action results. Both currently fall back to `full` with a warning. I always set `mode: "none"` because the post-action delta wasn't pulling weight; if `tree_diff` actually emitted just the changed subtree (or even just `appeared:`/`removed:` with refs), that would be the natural default.

### A3. `find({ contextRef, confidenceFloor })`

Two adds:

- **`contextRef: <ref>`** — limit ranking to descendants of that ref. "the seconds input under the the AI feature panel" without me having to encode the relationship in the natural-language query.
- **`confidenceFloor: 0.5`** — emit a `warnings: ["no candidate scored confidently above 0.5; consider raw selector / snapshot scan"]` when nothing crosses the bar. Today find() always returns 3–5 candidates regardless of how badly they match; a "no confident match" signal would tell me to stop reading the candidates and fall through to a snapshot scan immediately.

### A4. Inline halt-screenshot reference, not a separate Read

Today site-docs writes the halt screenshot to `docs/<flow>/halts/<step>.png` and the path appears in the error. To act on it, the agent does a separate `Read` (one extra turn + a few hundred tokens of metadata). If the halt error included the screenshot inline (base64) or, more realistically, a short auto-generated description ("editor side panel; the start-time-seconds input is greyed out (`disabled`)"), I'd absorb it in the same turn.

### A5. Console-error summarisation in `ActionResult.console`

A single React component stack-trace this round spanned ~50 lines (~1500 tokens) in one ActionResult. The information content: "1 prop-type warning, `notched` on a div ancestor, in the FeaturePanelEditor render path." Surfacing the *summary* with a `truncated_chars: 1430` flag, and a `console_read` for the full trace, would mirror the existing `network.requests omitted (count N > cap 10); call network_read for details` pattern.

### A6. Smarter `mode` defaults across action primitives

Default `mode` is `scoped_snapshot` (which falls back to `full`). I set `mode: "none"` on every call this session. A heuristic default — emit `tree_diff` when `structure.appeared/removed` is non-empty, else nothing — would make the typical case cheap without me having to remember.

## B. New tools / primitives

### B1. `eval({ expr, returnType: "json"|"void" })` — controlled JS-eval primitive

I needed `window.__siteDocs.capture()` (the SDK's session-capture trigger) **three times** across this session — rounds 3, 7, and 8. Each time required dropping a 30-line Playwright-over-CDP `.cjs` script in `/tmp` and shelling out via Bash. browxai has every action primitive *except* the simplest one.

Untrusted-page-content concerns apply to the *return value* (the agent treats it as page-controlled), but the *call* itself originates from the agent (trusted). Adding `eval` with a clear "the result is page-controlled" warning would close one of the most felt gaps.

Alternative narrower shape: `dispatch({ event, target })` — fire a custom event on a target, no return value. Covers the `__siteDocs.capture()` case (it's effectively "tell the page to do its capture") without giving general JS execution.

### B2. `screenshot({ ref, describe: true })` — emit alt-text alongside the PNG

Once the CDP-attached `bbox` issue is fixed (adoption-report 🟢 #4), cropping to a single element gives tiny PNGs. Pairing with an auto-generated description ("blue button with text 'Create recap script' inside a panel labelled 'the AI feature'") would let the agent skip vision-reading the image when it's just confirming "yes the button is there."

### B3. `flow_lint <flow.yaml>` — site-docs static checker

Catches the round-6 mistakes at write-time:

- annotation on a step whose target's selector doesn't appear in any `wait_for` of a *later* step (i.e. the target probably won't survive the action — suggest `annotation.target` override).
- `[data-foo="x"]` selector on a target where the snapshot showed multiple matches (suggest `:visible` / `:nth-match`).
- `wait_for: { selector: $X }` without `timeout_ms` on a step that's heuristically long-async (parent step is a click on a button containing words like "generate" / "create" / "process").
- `extends` chain depth > 3 (warn about per-run cost).

Deterministic, static, no live page needed; would have saved ~30–60 minutes across rounds 6–8.

### B4. `site-docs run --start-from <step-id>` + `--cdp` mode

`--stop-after` exists; the inverse `--start-from` does not. Combined with a hypothetical `site-docs run --cdp http://localhost:9222`:

- attach to the same Chrome the calibration is happening in
- skip the `extends` chain's already-completed steps
- replay only the new tail step you're iterating on

This collapses the round-7/8 inner loop from ~5 minutes per change (re-generate scripts every time) to ~5 *seconds* (page is already in the post-generate state; just re-run the one new step). The single biggest accelerant on long-async flows.

Risk: page state pollution between runs. Mitigations: explicit opt-in, or a `site-docs reset --to-step <step-id>` that re-walks just enough to restore state.

### B5. `await_human({ kind: "confirm" | "input" })` — already on the deferred list

I lost ~3–4 turns this session waiting for the user to type "Done!" / "Logged in!" — `confirm({prompt: "logged in?", default: yes, timeoutMs: 600000})` returning `{ok: true}` would close most of them.

### B6. Workspace co-location helper — `browxai init --consumer <workspace>`

Today the dual-registration recipe is hand-typed. `browxai init --consumer ~/site-docs/target-app-2 --test-attrs data-testid,data-type` could:

- create `<workspace>/.browxai/`
- register both MCP entries user-scope with the right env block
- detect target-app-like test conventions by sniffing the codebase (greppable: `data-type=` count vs `data-testid=` count in `src/`)
- print the right `BROWX_TEST_ATTRIBUTES` order

Removes the "did I get the order right" guesswork.

### B7. `browxai chrome [start|stop]` — own the `--cdp` Chrome lifecycle

Today the runbook tells the human to launch Chrome with the right flags. browxai *consumes* that Chrome but doesn't own it. A `browxai chrome start --persistent` subcommand that:

- uses the right flags every time (`--remote-debugging-port=9222 --disable-web-security --user-data-dir=$BROWX_WORKSPACE/chrome-profile`)
- has a stable profile location (so logins survive across browxai sessions)
- prints the URL so the user can navigate
- has `browxai chrome stop` for clean teardown

Reduces the Chrome-launch ceremony from a 4-flag command + a manual reset of `/tmp/site-docs-chrome` to a one-liner.

## C. Calibration-loop UX

### C1. Persistent named refs across snapshots

`ref=eN` is per-snapshot; after every action that mutates structure I have to re-find anchors I'd already located. A `browxai name_ref({ref, name: "feature-area_tab"})` that resolves the underlying selector and rebinds across snapshots — `browxai click({named: "feature-area_tab"})` — would let me carry a small set of anchors session-wide.

### C2. Calibration-walk → flow-file scaffold

Optional "recording" mode: `browxai start_recording({flow_name: "feature-9-edit-timing"})` → I do find/click/fill as normal → `browxai end_recording()` emits a draft `flow-file.yaml` with the locators (from the top-ranked find() candidate per step) and the action sequence. Agent reviews and edits.

Cuts ~15–20 minutes per flow of YAML hand-writing. Tradeoff: the agent loses the discipline of "deciding what to write"; but for the 70% case where the walk is straight-through, that's an acceptable tradeoff.

### C3. Annotation arrow nudge / fine positioning

`arrow: top|left|right|...` is coarse. When two annotations land in the same region (common when I want to call out two adjacent controls), they overlap. `arrow: { side: "top", nudge: { x: 20, y: 0 } }` — tiny ergonomic, big readability win.

### C4. `site-docs flow tree` — visualise an `extends` chain

Today I keep the chain in my head: `feature-6-translate extends feature-3-generate-scripts extends open-editor`. With 11 flows in the workspace and 3-deep chains, a `site-docs flow tree` that prints the parent/child graph + step-id-uniqueness check would help me spot inheritance mistakes before they bite at run-time.

## D. Cross-cutting

### D1. Shared "actionable" predicate between snapshot and run

The biggest correctness gap I hit twice: browxai's snapshot says an element is fine; site-docs's runtime halts on actionability. Both run on Playwright underneath. If `find()` exposed an `actionable: boolean | "disabled" | "covered" | "off-screen"` (alongside `stability` / `bbox`), the calibration agent would know at write-time that the start-time inputs are `disabled` and not bother fill'ing them. Same fix would have caught the round-6 hidden-duplicate boundingBox in the snapshot, before the click step even compiled.

This is the single highest-leverage change here — turns two of the asks in the adoption report into one design.

### D2. Pre-seeded test fixture in the target app

Repeated across the workspace report, the adoption report, and this one: a project in target-app that already has scripts + audio + translations would cut every downstream flow's runtime by ~3–15 min and remove the "first video varies day-over-day" risk. **Not a browxai/site-docs ask** — it's a target-app ask — but it's the largest single time-saver still on the table.

Adjacent: an "instant-generate" / stub mode on the target's BE for test runs (return canned scripts immediately). Even better.

### D3. `browxai doctor` — environment & connectivity check

Today before starting a run, the human verifies:
- browxai built (`dist/cli.js` exists)
- both MCP entries registered
- `--cdp` Chrome on :9222 reachable
- `BROWX_TEST_ATTRIBUTES` set with the right order
- `BROWX_WORKSPACE` writable

A single `browxai doctor` that prints all five with ✓/✗ and one-line fixes would replace the ~5 manual checks I did at the top of round 8.

## E. What I'd *not* prioritise

A short "no" list to keep the wishlist honest:

- **Replacing site-docs `run` with browxai entirely.** site-docs run is deterministic, no agent in the loop, no MCP. Adopting browxai there would buy nothing and lose the determinism. The split (browxai = discovery, site-docs = execution) is right.
- **Tier-3/4 selectorHints.** Useful in principle, but on target-app's heavy-`data-type` shape, tier-1 carried 90% of the load and tier-2 (`role+name`) caught most of the rest. Tier-3 (`text on stable role`) and tier-4 (`#id / semantic`) would round-trip a few cases at best.
- **A standalone session-wide `network_read` buffer.** The per-action `ActionResult.network` slice was sufficient for everything this session.
- **Generalising the agent-driven `/site-docs:calibrate` MCP-server skill.** The current "calibrate by hand following the playbook" workflow is fine and the few hundred lines of MCP plumbing wouldn't pay back unless someone needs to do this 10× a week.
- **Replacing the `--cdp` recipe with the legacy Claude-in-Chrome path.** Already deprecated in the runbook; mentioning only because it would be a tempting "small fix" — don't.

## F. Roughly-prioritised "if-I-had-one-week" ranking

If we picked the top three improvements that would change the most for the *next* adoption (any target, not just target-app), I'd pick:

1. **D1 — `actionable` on `find()` results.** Catches a class of failures at calibration-time, not at run-time. One change, surface area ~10–20 LOC, eliminates two of the three 🟡 asks in the adoption report.
2. **A1 — Scoped snapshots.** Halves token cost on the most frequent expensive operation. Probably ~50–100 LOC behind a `scope: <ref>` parameter on `snapshot()`.
3. **B4 — `site-docs run --start-from <step-id> --cdp`.** Collapses the `extends`-chain re-execution cost from ~3 min to ~3 sec. Requires a coordination contract between site-docs and the `--cdp` Chrome (state ownership), but pays for itself in ~4 iterations.

The rest are individually small wins; collectively they shave the rough edges off a workflow that already produces results.
